# Fix de errores — 2026-07-03 / 2026-07-04

## 1. `auth/login` 401: falta `centroId`

**Causa**: El usuario `jperez` tiene rol `Medico` en 3 centros. El backend exige `centroId` cuando el usuario no es administrador y tiene acceso a múltiples centros.

**Solución**: El frontend debe enviar `centroId` en el login. Ya funciona con:

```json
POST /api/v1/auth/login
{"username":"jperez","password":"123456","centroId":"00000000-0000-0000-0000-000000000001"}
```

**Usuario admin** (no requiere centroId):
- user: `admin` / pass: `admin` — creado por migration `014`, seed-data.sql lo setea como ACTIVO

## 2. `estructura-interna/tipos-documento` 403 Forbidden

**Causa**: El controller `EstructuraInternaController` tenía `[Authorize(Roles = "Administrador Seguridad")]` a nivel clase, restringiendo todos los endpoints solo a ese rol.

**Archivo**: `back/src/VitalFlow.His.Api/Controllers/EstructuraInternaController.cs`

**Fix**: Se movió la autorización a nivel de método:

| Endpoint | Roles permitidos |
|----------|-----------------|
| `GET tipos-documento` | Administrador, Administrador Seguridad, **Medico**, Auditor, Administrativo, Cajero, Enrolamiento Persona |
| `GET personas/busqueda` | Administrador, Administrador Seguridad, **Medico**, Auditor, Administrativo, Cajero, Enrolamiento Persona |
| `GET nodos` | Administrador Seguridad (sin cambio) |
| `GET {nodoId}/registros` | Administrador Seguridad (sin cambio) |
| `POST {nodoId}/registros` | Administrador Seguridad (sin cambio) |

**Ref**: Se quitó `[Authorize]` de clase y se agregó `[Authorize(Roles = "...")]` individual en cada método.

**Requerido**: Rebuild + deploy del backend (`docker compose build his_backend && docker compose up -d his_backend`).

**Localidades disponibles**: 24 provincias, ~240 localidades cargadas por migration `033_feature_ubicacion_provincias_localidades.sql`.

**Posibles soluciones**:
- **Corto plazo**: Verificar desde el frontend qué `localidadId` se está enviando y asegurar que exista en `sch_ubicacion.localidad`.
- **Mediano plazo**: Agregar manejo de excepción FK en `PostgresPersonaRepository.UpsertDomicilio()` para devolver 400 en lugar de 500.
- **Largo plazo**: Sincronizar catálogo de localidades entre frontend y backend.

**Localidades BA principales** (ya existen):
```
BA001 La Plata, BA002 Mar del Plata, ..., BA051 Ciudad Evita
```

**Localidades CABA** (ya existen):

## 7. `GetDomicilio()` / `GetContactos()` 500 en NULL fields

**Causa**: Se usaba `reader.GetString(ordinal)` directo sin `IsDBNull` check. Si cualquier columna era NULL, lanzaba `InvalidCastException`.

**Archivo**: `back/src/.../PostgresPersonaRepository.cs`

**Fix**: Agregado `reader.IsDBNull(ordinal)` en cada columna de:
- `GetDomicilio()` (columnas 3-12)
- `GetContactos()`, `CreateContacto()`, `UpdateContacto()` (columnas 2-9)
- Fecha_nacimiento (columna 6): usa `GetFieldValue<DateOnly>()` solo si no es NULL

## 8. Migrations 039 y 044 no idempotentes

**Causa**: `039` inserta prácticas con UUIDs hardcodeados sin `ON CONFLICT`. `044` inserta ~490 catálogos sin verificar existencia previa.

**Archivos**: `db/migrations/039_feature_import_practicas_glosario.sql`, `044_feature_practica_catalogo.sql`

**Fix**: 
- `039`: cada fila tiene `ON CONFLICT (id) DO NOTHING`
- `044`: todas las INSERTs envueltas en `DO $$ BEGIN IF NOT EXISTS (SELECT 1 FROM ...) THEN ... END IF; END $$;`

## 12. `personas/{id}/domicilio` PUT 500 → 400 BadRequest en FK violation

**Archivo**: `back/src/.../Controllers/PersonasController.cs:160-171`

**Causa**: El frontend envía `localidadId` que no existe en `sch_ubicacion.localidad`, el motor PostgreSQL lanza una excepción de FK (`23503`) que no era capturada, resultando en 500 Internal Server Error.

**Fix**: Agregado catch específico para `PostgresException` con `SqlState == "23503"` (foreign_key_violation) que devuelve 400 BadRequest con mensaje claro indicando que el `localidadId` no existe en el catálogo.

**Requerido**: Rebuild + deploy del backend (`docker compose build his_backend && docker compose up -d his_backend`).

## 15. `NpgsqlOperationInProgressException` al crear bloque (2026-07-04)

**Causa**: `RegenerateCupos()` en `PostgresAgendaRepository.cs` abría un `SqlDataReader` para leer los bloques activos y, dentro del `while (reader.Read())`, ejecutaba `cmd.ExecuteNonQuery()` para insertar cupos en la **misma conexión Npgsql**. Npgsql no permite operaciones concurrentes en una conexión — lanza `NpgsqlOperationInProgressException` cuando se intenta un segundo comando mientras un `DataReader` está activo.

**Stack trace**:
```
PostgresAgendaRepository.RegenerateCupos(Guid agendaId) line 1525  ← cmd.ExecuteNonQuery()
AgendaService.RecalcularCupos(Guid agendaId) line 1099
AgendaService.AddBloque(Guid agendaId, CreateBloqueRequest request) line 810
```

**Solución**: Separar la lectura de los datos de la escritura:
1. Leer todos los bloques activos en un bucle y construir una lista en memoria de `(BloqueId, HoraInicio, HoraFin)` — esto agota el `DataReader` y cierra la conexión para el siguiente comando
2. Fuera del bloque `using (reader)`, iterar la lista y ejecutar los INSERTs con `cmd.ExecuteNonQuery()`

**Archivo**: `back/src/.../Infrastructure/Agenda/PostgresAgendaRepository.cs:1459-1530`

**Fix aplicado**:
```csharp
// Antes: INSERT dentro del while(reader.Read()) — CRASH
using var reader = cmdBloques.ExecuteReader();
while (reader.Read())
{
    // ... leer datos ...
    using var cmd = new NpgsqlCommand(sqlInsert, conn);
    // ...
    cmd.ExecuteNonQuery();  // 💥 NpgsqlOperationInProgressException
}

// Después: leer primero, insertar después
var slots = new List<(Guid, DateTimeOffset, DateTimeOffset)>();
using var reader = cmdBloques.ExecuteReader();
while (reader.Read())
{
    // ... leer datos, agregar a slots ...
}
// reader cerrado aquí
foreach (var (bloqueId, inicio, fin) in slots)
{
    using var cmd = new NpgsqlCommand(sqlInsert, conn);
    cmd.ExecuteNonQuery();  // ✅ conexión libre
}
```
